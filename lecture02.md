## 1.バージョン管理システム

### 代表的なもの
**SVN**
- 中央集権型
- サーバーは自分たちがセットアップ
　問題: サーバーが壊れた時、復旧までの間、仕事ができない

**Git**
- 分散型
- サーバー運用してくれている(Git-Hub等のサービス)
- 自分の中で完結するため、サーバーが壊れても仕事への影響は最小化
- チェンジセット(変更箇所のみ)を保存するため、ブランチが切りやすい


## 2.Git
自分のPC ⇒ ローカルリポジトリ<br>
サーバー ⇒ リモートリポジトリ<br>

### コマンドの種類
| Command | 機能 |
|---|---|
| Clone | リモートリポジトリのコピー |
| Commit | ローカルリポジトリへの反映 |
| Push | リモートリポジトリへの反映 *1|

*1: Commit後の内容がまとめて反映(履歴ごと)

#### ご参考
CommitとPush同時にできる。同時でも困ることはない。

### 作法
- ブランチ作成(枝分かれ)してから開発開始する
- 運用ルールは開発チーム次第

### GitHubのアカウント
アカウントは、ずっと使うことになる想定でアカウント名を決めること<br>
⇒サービス(無料/有料)の考え方が変遷する。<br>
  都度状況確認した方がよい

## 3.Markdown
Gitで扱う説明書、ドキュメントはMarkdownで書ける
Slakでも使える<br>
⇒Markdown 簡単なものは習得しておくとよい

## 4.チーム開発におけるバージョン管理
チームメンバーが個別に作業し、最終的にレビューをパスしたものをマージして取り込む<br>
運用方法には、Git-Flow, GitHub-Flowがある<br>
⇒GitHub-Flowがシンプル (main先行)

### ブランチの切り方(種類)
- main: デフォルトブランチ名(動作保証された状態)
- feture: 機能追加
- bugfix: 修正

## 5.開発で役立つ GitHub TIPS
### Pull Request
機能が実現できる最小単位で出すのが望ましい<br>
⇒ファイル変更数が膨大な場合は、レビュアーが大変 *2<br>
*2: 変更ファイル数に気をつけること

### 承認機能
- チームメンバーの能力がバラバラの場合はあった方がよい
- 能力が同等の場合は、スピード重視し承認機能を入れないこともある
- 承認に強い権限を入れることもできる(マージさせない)
